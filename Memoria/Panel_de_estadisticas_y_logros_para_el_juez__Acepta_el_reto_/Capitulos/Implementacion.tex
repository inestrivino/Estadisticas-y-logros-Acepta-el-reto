\section{Entorno de desarrollo y herramientas}

Para el desarrollo de la aplicación se ha empleado un entorno basado en tecnologías web modernas orientadas al desarrollo de aplicaciones distribuidas y escalables. A continuación, se describen los lenguajes, frameworks y herramientas utilizadas durante la implementación del proyecto.

En cuanto a los lenguajes de programación y marcado, se utilizaron TypeScript\footnote{\url{https://www.typescriptlang.org/}} en su versión 5.9.3 o superior y JavaScript como soporte para determinadas librerías y configuraciones del ecosistema Node.js. Para la construcción de la interfaz de usuario se empleó React 19.2.2 o superior, permitiendo el desarrollo de componentes reutilizables (lo cual es útil para una futura integración completa de la aplicación con \aer, ya que no sería necesario recrear toda una interfaz) y una gestión eficiente del estado de la aplicación. Además, se utilizaron HTML5 y CSS3 para la estructuración y el diseño visual de la interfaz.

El entorno de ejecución del backend se basó en Node.js 12.17\footnote{\url{https://nodejs.org/en}} o superior, junto con npm 7\footnote{\url{https://www.npmjs.com/}} o superior como gestor de dependencias y paquetes. Para el almacenamiento temporal y la gestión de caché se utilizó Redis 5.8.3\footnote{\url{https://redis.io/}} o superior, mejorando el rendimiento de determinadas operaciones del sistema gracias a ser una base de datos dirigida a alta eficiencia. Asimismo, se incorporó RabbitMQ \footnote{\url{https://www.rabbitmq.com/}} como sistema de mensajería, permitiendo la comunicación asíncrona entre servicios y facilitando una arquitectura desacoplada.

La contenerización de los distintos servicios se realizó mediante Docker 20.10\footnote{\url{https://www.docker.com/}} o superior, aunque durante el desarrollo se utilizó la versión 29.2.1. Esta versión incluye soporte integrado para Docker Compose\footnote{\url{https://docs.docker.com/compose/}}, herramienta empleada para la orquestación de los contenedores y la definición de los distintos servicios de la aplicación. Las imágenes Docker utilizadas para los servicios basados en Node.js se construyeron sobre imágenes \texttt{node:alpine}, debido a su reducido tamaño y eficiencia.

En el entorno de producción del frontend se utilizó Nginx\footnote{\url{https://nginx.org/}} como servidor web, principalmente por su alto rendimiento y velocidad en el servido de contenido estático y renderizado de la aplicación cliente.

Para el control de versiones del proyecto se empleó Git\footnote{\url{https://git-scm.com/}} junto con la plataforma GitHub\footnote{\url{https://github.com/}}, facilitando la gestión del código fuente, el trabajo colaborativo y el seguimiento de cambios. Finalmente, el entorno de desarrollo principal utilizado fue Visual Studio Code (VSCode), gracias a su integración con el ecosistema JavaScript/TypeScript y sus herramientas de depuración y automatización.

En comparación con la arquitectura tradicional descrita para \aer\ en \cite{Pedro_Pablo_Gomez_Martin_Marco_Antonio_Gomez_Martin_2017}, basada en Tomcat, PHP y MySQL, la solución desarrollada en este trabajo adopta un enfoque más moderno. El uso de las tecnologías elegidas permite mejorar aspectos como la escalabilidad, la modularidad y el rendimiento, a la vez que la arquitectura desacoplada sigue permitiendo integrar el sistema de estadísticas y logros dentro de \aer sin tener que realizar cambios en el OJ.

\section{Arquitectura del sistema}

\subsection{Patrones de diseño del \textit{backend}}
\label{sec:patronesDiseno}

El \textit{backend} organiza una lógica de negocio en torno a cuatro grupos de componentes especializados: actualizadores de estado, registradores, estadísticas de experiencia y logros. En todos ellos cada grupo define un contrato común (interfaz o clase abstracta) y el servicio que los gestiona delega el trabajo iterando sobre una colección de implementaciones intercambiables, lo que permite incorporar nuevos comportamientos sin modificar el código existente.

\subsubsection{Actualizadores}

Los actualizadores calculan y mantienen el estado de usuarios y problemas a medida que se procesan envíos. Cada actualizador encapsula la lógica de una estadística concreta.

En este grupo se aplica el \textbf{patrón Template Method}\footnote{Más información sobre este patrón: \url{https://refactoring.guru/design-patterns/template-method}}: la clase abstracta define el flujo del algoritmo de inicialización en un método concreto, que a su vez invoca métodos abstractos que cada subclase implementa de forma específica. El método de actualización, también abstracto, es invocado por el servicio para cada envío procesado. El diagrama de clases correspondiente se muestra en la figura~\ref{fig:patrones-actualizadores}.

\subsubsection{Registradores}

Los registradores persisten en Redis el estado calculado por los actualizadores. Cada registrador implementa una interfaz común que define las operaciones de escritura y borrado.

El servicio itera sobre la colección de registradores y cada uno persiste el fragmento del estado que le corresponde. El diagrama de clases se muestra en la figura~\ref{fig:patrones-registradores}.

Se aplica el patrón Strategy: el servicio de experiencia mantiene una lista de estadísticas y las agrega para obtener la XP total del usuario. El diagrama de clases se muestra en la figura~\ref{fig:patrones-estadistica-experiencia}.

\subsubsection{Logros}

Los logros comprueban si un usuario ha alcanzado una condición determinada. Todos implementan una interfaz común que, entre otros campos, expone el método que evalúa dicha condición.

También se aplica el patrón Strategy: el servicio de logros delega en cada instancia la evaluación de su condición particular. Las implementaciones más complejas son clases parametrizadas que reciben sus umbrales por constructor, mientras que los logros más simples se implementan como objetos literales. \section{Desarrollo del \textit{backend}}

La pieza central del desarrollo del \textit{backend} es el proceso de carga inicial de envíos, encargado de obtener, parsear y procesar el histórico completo de envíos registrados en \aer\ de forma eficiente y tolerante a interrupciones. Junto a este proceso, se implementó un mecanismo de actualización selectiva basado en versiones que permite incorporar nuevas estadísticas, modificar logros o recalibrar la experiencia sin necesidad de reprocesar la totalidad de los datos históricos.

\subsection{Carga inicial de envíos}

El proceso de carga inicial constituye la operación más significativa del \textit{backend}, ya que transforma los datos crudos del juez en estadísticas, logros y puntuaciones de experiencia de los usuarios. Este proceso se activa al arrancar el sistema y está diseñado para poder reanudarse en caso de interrupción, gracias al sistema de \textit{checkpoints} descrito en la siguiente sección.

Para localizar el punto de inicio en la paginación de la API externa, se aplica primero una búsqueda de saltos exponenciales que acota el intervalo de páginas relevante y, a continuación, una búsqueda binaria que identifica la página exacta desde la que comenzar. Una vez determinado ese punto, los envíos se procesan en bloques de 50 páginas con un \textit{pool} de 20 peticiones paralelas, lo que permite aprovechar la concurrencia sin saturar la API. Al finalizar cada bloque se persisten los resultados en Redis y se emite un evento de progreso al cliente.

El proceso está compuesto por varios servicios, cada uno con una responsabilidad claramente delimitada: la orquestación de la carga, la gestión del estado de progreso, el mantenimiento de los \textit{checkpoints} por componente, la acumulación de estados en memoria, la evaluación y persistencia de logros, y el cálculo de experiencia. Esta separación de responsabilidades facilita tanto el mantenimiento como la extensión del sistema, ya que cada servicio puede evolucionar de forma independiente.

Un aspecto clave del diseño es que cada estadística y cada logro mantiene su propio \textit{checkpoint}, de forma que el progreso de cada uno se registra por separado. Esto cobra especial importancia cuando se necesita recargar algún componente: si una estadística lleva procesada la mitad del histórico y en ese momento se decide incorporar otra nueva, el sistema puede reiniciarse sin perder el trabajo ya realizado. Al reanudar, la estadística nueva comienza desde el principio, mientras que la que ya estaba en curso retoma desde su \textit{checkpoint}. Cuando la nueva alcanza ese punto ambas avanzan juntas, de modo que nunca es necesario reprocesar más de lo imprescindible.

\subsection{Actualización selectiva mediante versiones}

Un requisito fundamental del sistema es poder incorporar nuevas estadísticas, modificar el comportamiento de las existentes o redefinir logros sin tener que reprocesar la totalidad de los envíos históricos. Para satisfacerlo se implementó un mecanismo de versiones que se ejecuta al arrancar el servidor, cuya secuencia se ilustra en la Figura~\ref{fig:secuencia-arranque}.

Cada calculador de estadísticas y cada logro dispone de un número de versión declarado en el código. Al arrancar, el sistema compara estas versiones con las almacenadas en Redis y actúa de forma selectiva según lo que haya cambiado: las estadísticas modificadas se recalculan desde cero sin afectar al resto, los logros cuya definición haya variado se reevalúan junto con las estadísticas de las que dependen, y la experiencia se recalcula automáticamente siempre que cambie cualquier componente que contribuya a su cómputo. Además, es posible forzar el recálculo de la experiencia de forma independiente mediante una variable de entorno, sin necesidad de modificar ningún componente.

De este modo, añadir una estadística nueva, ajustar los umbrales de un logro o modificar el peso de algún factor en el cálculo de experiencia no requiere intervención manual sobre los datos almacenados: basta con incrementar el número de versión del componente afectado para que el sistema detecte el cambio en el siguiente arranque y reconstruya únicamente lo necesario.

En la práctica, el mecanismo se activa de dos formas. Para un reinicio total basta con incrementar la variable de entorno \texttt{VERSION}, lo que provoca que el sistema borre completamente Redis al arrancar y reconstruya todos los datos desde cero. Para recargar un componente concreto basta con incrementar su número de versión en el código: las estadísticas y los logros pueden resetearse de forma independiente, y la experiencia se recalcula automáticamente cuando el sistema detecta que alguno de los componentes de los que depende ha cambiado. Si se desea forzar el recálculo de la experiencia sin modificar ningún otro componente, basta con incrementar la variable de entorno \texttt{XPVERSION}.

\section{Desarrollo del \textit{frontend}}

El desarrollo del \textit{frontend} comenzó con una fase de investigación en la que se identificaron y evaluaron las tecnologías necesarias para cubrir los requisitos del proyecto: la construcción de componentes de interfaz, la gestión del estado, la visualización de datos y la comunicación en tiempo real con el servidor. A partir de esa investigación se fijaron las dependencias principales (React, Recharts\footnote{\url{https://recharts.github.io/}} y Socket.io\footnote{\url{https://socket.io/}}, entre otras) y se estableció la estructura inicial del proyecto.

Con ese punto de partida, el desarrollo arrancó siguiendo una arquitectura clásica multipágina, en la que cada vista correspondía a una ruta y una página independiente. Esta elección respondía también al momento del proyecto: el equipo de desarrollo estaba cursando asignaturas de desarrollo web, y se trasladaron al proyecto los conceptos y técnicas que se aprendían en paralelo. Sin embargo, a medida que avanzaba el desarrollo con React se hizo evidente que continuar por esa vía desembocaría en una arquitectura híbrida, mezcla de enrutamiento multipágina y componentes React, sin las ventajas plenas de ninguno de los dos enfoques. Para evitar quedarse a medias entre dos arquitecturas incompatibles se tomó la decisión de dar el paso completo a una SPA (\textit{Single Page Application}). Esta transición requirió una investigación adicional sobre el enrutamiento del lado del cliente y la gestión del estado global con la Context API de React. El resultado final es una SPA estructurada en cinco vistas principales: la página de inicio, las estadísticas de usuario, el medallero de logros, las estadísticas de problema y la tabla de clasificación.

A lo largo del desarrollo también se introdujeron ajustes sobre los \textit{wireframes} iniciales. La vista del medallero de logros añadió el tratamiento visual de los logros sorpresa: su descripción aparece con un efecto de desenfoque hasta que el usuario los desbloquea, reforzando la sensación de descubrimiento descrita en el diseño del sistema de gamificación. La barra lateral (\textit{sidebar}) se adaptó progresivamente para soportar distintos tamaños de pantalla, requiriendo varias iteraciones hasta conseguir un comportamiento coherente en dispositivos móviles. La tabla de clasificación fue rediseñada para incluir filtros por nivel de dominio y paginación, funcionalidades que no estaban contempladas en el \textit{wireframe} original pero que mejoraban la navegabilidad y la experiencia de uso. \subsection{Componentes principales}

El \textit{frontend} se estructura en torno a un conjunto de componentes reutilizables que encapsulan tanto la lógica de presentación como la de interacción.

El buscador es el punto de entrada principal a las estadísticas: el usuario escribe el nombre de un usuario o problema de \aer\ y el componente consulta la API en tiempo real y muestra las coincidencias en un desplegable. La última selección se persiste en \texttt{localStorage} para recuperarla automáticamente en la siguiente visita. Este componente se construyó sobre una plantilla reutilizable que lo integra en cada vista como elemento de cabecera, de forma que el usuario puede cambiar entre vistas manteniendo el usuario y el problema seleccionados.

La etiqueta de nivel muestra el nivel actual del usuario con un color asociado a cada rango. Las tarjetas de logro muestran el icono, el nombre y la descripción de cada logro; en el caso de los logros sorpresa, la descripción aparece con un efecto de desenfoque hasta que el usuario los desbloquea.

Las estadísticas se presentan mediante una combinación de componentes numéricos y gráficos. Los valores individuales se muestran a través de tarjetas reutilizables que admiten distintas variantes visuales según el contexto. Para los datos con distribución porcentual, como la proporción de veredictos o los lenguajes utilizados, se emplearon diagramas de sectores. La evolución de la experiencia a lo largo del tiempo se representa mediante un gráfico de líneas mensual. Ambos tipos de gráfico se implementaron con la librería Recharts. Todos los componentes se suscriben a eventos del servidor mediante Socket.io, de forma que cualquier actualización de datos se refleja automáticamente en la interfaz sin necesidad de recargar la página. Con el objetivo de evitar saltos bruscos durante la carga de datos, se implementaron componentes de \textit{skeleton loading}: marcadores de posición animados que replican la estructura visual del contenido y se sustituyen por los datos reales una vez disponibles.

\section{Despliegue y mantenimiento (CI/CD)}

El proyecto incorpora un flujo de integración y despliegue continuo (CI/CD) orientado a facilitar el desarrollo colaborativo, la validación automática del código y el despliegue de la aplicación en distintos entornos. Para ello, se desarrollaron distintos scripts y configuraciones que automatizan tanto la ejecución de pruebas como la gestión de contenedores y procesos de compilación.

En relación con la integración continua, el proyecto dispone de un flujo automatizado definido en el archivo \texttt{ci.yml}, encargado de ejecutar pruebas y comprobaciones automáticas sobre el código fuente antes de integrar cambios en las ramas principales. Los detalles concretos sobre las pruebas realizadas y su evaluación se describen en el capítulo correspondiente \ref{cap:evaluacion}. Además, se dispone de una página de monitorización que permite supervisar el estado y funcionamiento de los distintos servicios de la aplicación según los tests.

Durante el desarrollo se detectaron determinadas limitaciones relacionadas con el uso de contenedores Docker en entornos locales, por lo que se implementó un script específico denominado \texttt{startDev.js}. Este script automatiza el arranque del entorno de desarrollo y simplifica la gestión de los servicios necesarios para ejecutar la aplicación correctamente.

Asimismo, el entorno de desarrollo permite la depuración directa de los contenedores Docker gracias a la configuración incluida en \texttt{.vscode/launch.json}. Esto facilita la identificación y resolución de errores tanto en el frontend como en el backend sin necesidad de ejecutar los servicios fuera de los contenedores.

La aplicación se distribuye mediante contenedores Docker y utiliza distintas configuraciones según el entorno de ejecución. En producción, el comando \texttt{npm start prod} lanza un entorno basado en Docker Compose que, previamente al despliegue, compila tanto el frontend como el backend transformando el código TypeScript en JavaScript optimizado para ejecución en producción. Este proceso permite reducir dependencias innecesarias y mejorar el rendimiento y la portabilidad de la aplicación desplegada.

En cuanto a la gestión de versiones, se siguió una convención de versionado semántico, considerando la finalización del proyecto como la versión \texttt{1.0}. Para el desarrollo colaborativo se adoptó un flujo de trabajo basado en ramas para cada \textit{feature} que se desea implementar: un (\textit{branching workflow})\footnote{Puede leerse más sobre esta estrategia de organización y despliegue en el siguiente enlace: \url{https://git-scm.com/book/en/v2/Git-Branching-Branching-Workflows}}. La rama \texttt{main} contiene siempre la versión estable y desplegable de la aplicación, mientras que la rama \texttt{dev} se utiliza para la integración y validación de nuevas funcionalidades. Cada nueva característica se desarrolla en una rama independiente, permitiendo aislar cambios y reducir conflictos. Adicionalmente, en determinadas ocasiones se crearon ramas intermedias destinadas a realizar fusiones y comprobaciones previas sin comprometer la estabilidad del código principal.
